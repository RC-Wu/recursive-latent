## 1. User Priorities For This Cycle

### 1.0 Strict Matched-Task Correction

The user rejected the previous loose "similar-looking candidate" baseline matching. The correct protocol is now:

- Start from a concrete traditional procedural task, not from our existing pretty result.
- Preserve the task category and recursive/growth mode as closely as possible:
  - if L-system produces a pine/tree canopy, PS-RSLG must produce a pine/tree-canopy-like recursive branching asset;
  - if space colonization produces an attractor-filled crown or root field, PS-RSLG must use analogous attractor/competition controls;
  - if DLA produces a coral/crystal/frontier accretion, PS-RSLG must use a frontier-attachment/occupancy-exclusion rule, not an unrelated coral-looking mesh;
  - if IFS produces a transform-copy branching/lattice/growth pattern, PS-RSLG must use the corresponding transform family.
- For every matched task, record the operator composition explicitly and reproducibly:
  `root source -> rule family -> control parameters -> projection/naturalization schedule -> texture/PBR guide -> render protocol`.
- Generate multiple roots/seeds/operator settings for each family and select the strongest case only after visual and metric screening.
- Traditional baselines must be diverse enough: not only one branch or one DLA cluster. At minimum include pine/canopy, root/vine, bush/shell, coral/frontier, crystal/lattice, and transform/fractal tree/ornament tasks.
- All final comparison visuals must be mesh/PBR renders. Matplotlib or point-cloud previews are invalid as paper evidence.
- Render protocol correction:
  - upright and centered assets;
  - square pure-white raw overview saved separately;
  - camera-level zoom renders, not just 2D crop, with the zoom view rendered at full resolution;
  - zoom panels square and same displayed size as the overview;
  - rectangular callouts on the source view should select informative detailed regions such as branch tips, leaf/needle clusters, crown junctions, crystal facets, coral tips, or DLA attachment necks, not uninformative trunk-only regions;
  - use nested/multiscale zoom where depth is a method claim.

### 1.1 Masked Local Naturalization Ablation

The user accepts local masked naturalization as the right direction but requires a stable, attractive, main-paper-ready ablation. This must have both visual and quantitative support.

Required ablation columns:

1. Rule scaffold / no naturalization.
2. Global flow or global repair baseline.
3. Weak or final-only naturalization/projection.
4. Per-depth projection without masked naturalization.
5. Per-depth projection plus masked local naturalization.

The exact columns may be adjusted if implementation evidence shows a cleaner protocol, but the final figure must isolate the claim: local masked naturalization improves local surface/material plausibility while preserving grammar topology better than global repair.

Primary metrics:

- Connectivity/topology: connected components, largest-component ratio, root-attached mass, orphan mass, component growth by depth, attachment survival.
- Mesh quality: face components, non-manifold edge count, watertight/repair status, degenerate faces, normal consistency, triangle aspect ratio, surface roughness/Laplacian energy, axis-aligned blockiness score.
- Locality/preservation: anchor-region Chamfer/IoU before vs after naturalization, edit-mask-only change mass, handle/tip survival.
- Visual/semantic: multi-view CLIP/DINO similarity to class prompt and material prompt where available; render success; human-screening label for main-paper candidacy.

Candidate cases:

- Organic: vine/root or tree branch case with strong existing result.
- Frontier: coral/DLA-like case, only if connectivity can be preserved.
- Symmetry/hard surface: pyrite lattice or scifi lattice, where transform consistency can be measured.

### 1.2 Transform / Equivariance / Grammar Design

Transform results should not be overstated as strict Trellis2 equivariance. The current likely story is:

- Transform-copy is a grammar operator family.
- Empirical transform diagnostics define which transforms are safe enough to become reusable grammar rules.
- Good result: pyrite lattice / group-like repeated transform case.
- Weak result: bismuth/porous/DLA or aggressive radial cases, where copies remain visually or topologically unstable.

The paper should use good-vs-bad transform diagnostics to justify a grammar design choice: group-closed or lattice-like operators are admitted when projection and orbit metrics show stable support; unstable transforms are treated as boundary cases or appendix diagnostics.

### 1.5 Five-Column Evaluation Clarification

The user asked what the proposed five columns mean. The intended comparison is:

1. Classical baseline: native traditional algorithm.
2. Direct sparse grammar: our rule proposals with no projection/naturalization; shows raw failure mode.
3. Final-only cleanup/projection: shows why cleaning at the end is insufficient.
4. Per-depth projection: our structural core; demonstrates recursive-state stability.
5. Per-depth projection plus masked local naturalization plus texture/PBR: final asset-quality output.

If a case lacks a stable naturalization implementation, columns 4 and 5 should be separated: column 4 for structural claim, column 5 for export/appearance candidate only.
